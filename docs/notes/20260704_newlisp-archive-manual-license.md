# newLISPプログラミング言語の公式サイト閉鎖に伴う現存資料、ソースコード、マニュアル、およびライセンスに関する調査報告書

かつて新興のスクリプト言語として注目を集めたnewLISPの公式ドメインである `newlisp.org` は、現在オフラインの状態にあり、新規のアクセスが不可能な状況となっている 。newLISPは10.7.5（2019年5月12日リリース）をもって安定版のリリースを休止しているが、その独自のメモリ管理モデルや極めて軽量な実行環境は、組み込みシステムやスクリプト自動化の分野において現在も一定の技術的価値を有している 。公式ポータルの閉鎖という事態に直面しているものの、ソースコード、リファレンスマニュアル、周辺ツール、およびコミュニティの知見といったエコシステム全体は、世界中の分散型リポジトリやオープンソースアーカイブによって完全に保全されている 。本報告書は、現存するこれらのリソースの所在を体系的に明らかにし、そのライセンス条件と技術的特異性について、エンジニアリングおよび歴史的保存の観点から詳細に解説する。

## ソースコードの現存状況と入手経路

newLISPのプライマリ開発は、歴史的にSourceForge上でルッツ・ミュラー（Lutz Mueller）によって管理されており、公式のリリースビルドやアーカイブはすべてこのプラットフォーム上にアップロードされていた 。公式ドメインの閉鎖後も、SourceForgeのプロジェクトページは稼働を続けており、開発者が公式のソースコード（`newlisp-10.7.5.tgz` など）やWindows向けのインストーラーをダウンロードするための直接的な一次ソースであり続けている 。

これと並行して、モダンな分散型バージョン管理システム（DVCS）への移行とソースコードの追跡性の向上を目的として、有志による非公式のGitミラーが構築されてきた 。その代表例がGitHub上で公開されている `kosh04/newlisp` リポジトリである 。このリポジトリはSourceForgeから直接ソースファイルをフェッチし、開発者がコードベースのコミット履歴を詳細に追跡し、修正パッチを容易に適用できるように設計されている 。

このGitミラーリポジトリの言語構成比率は、newLISPがC言語で実装された軽量なコアエンジンと、その上で動作するnewLISPスクリプトのライブラリによって構成されているという構造的特徴を如実に表している 。

| --- | --- | --- |
| **C** | 59.1% | インタプリタのコアロジック（`newlisp.c`、`newlisp.h`、`nl-filesys.c`、`nl-import.c`、`nl-list.c` など） 。

 |
| **NewLisp** | 34.7% | 拡張モジュール、プラットフォーム別サンプルコード、テストスイート（`qa-specific-tests`、`modules` など） 。

また、このリポジトリは極めて多用なプラットフォームをターゲットとしたビルド用の Makefile を内包しており、移植性の高さを証明している 。サポートされている主要なビルドターゲットには、Linux、Darwin/Mac OS、各種BSD、SunOS、Cygwin、MinGW、Tru64、AIX、OpenSolaris、さらにはRaspberry PiやEmscripten/WebAssembly、歴史的なOS/2環境までもが含まれている 。

さらに、ミラー元の `master` ブランチは公式のSourceForgeソースを忠実に再現している一方、`develop` ブランチではGitHub固有の自動化設定（Travis CIやAppVeyor、Codecovといった継続的インテグレーションやテスト、Gitterチャット連携）が統合されており、コミュニティ主導による品質保証と近代化が進められている 。

## 開発用補助リポジトリおよび周辺ツール

newLISPの動作環境やスクリプティングを支援するための周辺ツールも、GitHubを中心とした開発者アカウントによって多数ホストされている。

### エディタサポートと統合開発環境

開発環境の利便性を高めるためのエディタプラグインや、専用の統合開発環境（IDE）が今なお利用可能である。Emacs向けには、シンタックスハイライト、キーワード補完、インタラクティブプロセス実行などをサポートする `kosh04/newlisp-mode` や、SLIMEとの連携を可能にする `swank-newlisp.lsp` が提供されている 。また、Atomエディタ向けに構築された `cormullion/language-newlisp` は、正規表現ベースでnewLISPの構文定義を行い、マニュアルに準拠した引数の記述順を確認できるスニペット機能などを提供している 。さらに、macOSおよびWindowsで動作する専用IDEである `newIDE` は、シンタックスハイライト対応のエディタ、統合型REPL、スタンドアロンバイナリへのビルド機能、そして検索可能なnewLISPマニュアルを同一パッケージ内に統合しており、包括的な開発環境を提供する 。

### 実装ライブラリと実用ツール

システムの操作や外部インターフェースとの統合のためのライブラリも現存している。Windows環境下でWin32 API（`user32.dll` や `gdi32.dll` など）をnewLISPから直接呼び出し、ネイティブGUIを構築するためのライブラリである `crossbowerbt/NewlispWin32Api` がその一例である 。また、Linux等でのGUI構築を容易にする `JeysonFlores/newLisp-GTK` バインディングも開発されている 。Cormullion氏による `cormullion/newlisp-projects` には、長年にわたり記述された多様なコード群がブログアーカイブとともに保存されており、実用的なスクリプティングの事例として有用である 。さらに、`oubiwann/newlisp-scripts` は、GNU makeとnewLISPに組み込まれた `getopts` ライブラリを利用し、無線接続（`wifi`）、バッテリーステータス（`bat`）、画面輝度（`backlight`）、複数Gitプロジェクト管理（`mgit`）、音量制御（`vol`）などを統合する軽量なコマンドラインユーティリティを提供している 。

## マニュアルおよび教育用ドキュメントの現存状況

newLISPを学習、あるいは実務に適用するうえで不可欠な技術ドキュメント類も、分散アーカイブを通じて安全に保護されている。

### テキスト形式リファレンスマニュアル

従来 `newlisp.org` の公式ダウンロードディレクトリに配置されていたHTML形式の公式マニュアルは、`cmpitg/newlisp-files` や `kosh04/newlisp-files` リポジトリの内部に `newlisp_manual.txt`（バージョン10.3.2ベース）として完全なプレーンテキストで収録されている 。このテキストドキュメントはブラウザでの読み込み負荷を低減し、ローカル環境へのダウンロードやコマンドラインからの検索が容易となるよう最適化されている 。

### 入門書「Introduction to newLISP」のオンラインおよびPDF版

体系的な学習ドキュメントとして高く評価されているWikibooksの『Introduction to newLISP』は、現在もWeb上で閲覧可能である 。また、これを印刷用にコンパイルした完全なPDF書籍（`Introduction_to_newLISP.pdf`）がWikimedia Commonsのファイルサーバー上に配置されている 。このドキュメントは、Dirk Hünniger氏のオープンソースプログラムによってLaTeXソースから自動生成されたものである 。本書にはnewLISPの基礎から応用までが詳細に記述されているが、特に注目に値するのは、以下の「3つの基本原則」に則った極めてシンプルな言語文法構造の解説である 。

* **原則1（リストの構造）：** リストとは、丸括弧によって囲まれた一連の要素の並びである 。
* **原則2（先頭要素の特異性）：** リスト内の最初の要素は特別な意味を持ち、常に関数やマクロなどの実行命令として評価される（ネストされたリストの評価や、クォートによる評価の抑制を伴う） 。
* **原則3（クォートによる保護）：** シンボルや式の前に配置されたクォートは、インタプリタに対してその式そのものを評価せずデータとして扱うよう指示する 。

## ライセンスおよび商標権の法的位置づけ

newLISPの技術資産を自社プロジェクトやフォークしたコミュニティ活動に転用するにあたっては、ライセンスおよび商標権の法的定義を正確に理解しておく必要がある。

### GNU General Public License v3 (GPLv3+)

newLISPの処理系本体、およびコアライブラリ群はすべて **GPLv3（GNU General Public License version 3、またはそれ以降のバージョン）** の下でリリースされている 。このライセンスに基づき、第三者が自由にソースコードを複製、修正、および再配布することが法的に完全に認められている。ただし、コピーレフト条項が適用されるため、newLISPを改変した派生エンジンを配布する場合には、その派生物のソースコード自体もGPLv3の下で公開しなければならないという義務が生じる 。なお、Emacs用編集モードである `newlisp-mode` などの周辺コミュニティツールについても同様にGPLv3が適用されている 。

### 商標権（Trademarks）による制約

一方で、知的財産のすべてが無条件で自由に使用できるわけではない。リポジトリの法的文書の規定において、"newLISP" および "Nuevatec" という名称およびロゴは、すべて開発者であるルッツ・ミュラー氏の登録商標として保護されている 。したがって、もしコミュニティが独自の互換インタプリタや改変ビルドを大規模に展開する場合は、ミュラー氏の商標権を侵害しないよう、異なるプロジェクト名（商標）を冠してリリースするなどの配慮が必要となる。

## 技術的特徴、パラダイム、およびコミュニティにおける評価

newLISPは、伝統的なLisp（Common LispやSchemeなど）の標準的な実装アプローチとは一線を画した設計思想を有しており、これが賛否を呼ぶ要因となっている 。

### 単一参照限定（One Reference Only: ORO）メモリ管理

伝統的なLispがヒープ領域に変数を配置し、到達不可能なオブジェクトをバックグラウンドで走査するマーク＆スイープなどのガベージコレクション（GC）を採用するのに対し、newLISPは「ORO（One Reference Only）」と呼ばれる独自の自動メモリ管理システムを採用している 。これは、変数が所属する「コンテキスト」のみが該当のオブジェクトを参照し、コンテキスト自体がグローバルに管理されるモデルである 。

このモデルでは、データ構造へ要素を格納する際や、関数へデータを引き渡す際に、原則として参照渡しではなく「コピー（ディープコピー）」が行われる 。この仕組みにより、システム内で循環参照が発生する可能性自体を根本から排除している 。これはC言語におけるスタック上のローカル変数管理に極めて近く、評価の階層が終了する（関数が呼び出し元に戻る）のと同時に、メモリ空間がスタックのように自動的に解放される構造になっている 。結果として、複雑なGCアルゴリズムを排したことで、約250KB〜300KBという極小のバイナリサイズと、瞬時の起動速度、およびリアルタイム処理に近い予測可能かつスムーズなメモリ回収動作を実現している 。

### 動的スコープとコンテキストによる名前空間

newLISPは静的（レキシカル）スコープを極力排除し、意図的に「動的スコープ（Dynamic Scoping）」を採用している 。関数は実行時に呼び出し元の全変数を参照可能であり、これにより極めて高度な柔軟性を確保している 。一方で、大規模開発における変数名の競合を防ぐため、「コンテキスト（Context）」と呼ばれるカプセル化された名前空間を利用する 。このコンテキストはプロトタイプベースのオブジェクト指向プログラムを可能にする枠組みとしても機能する 。

### コミュニティにおける批判と再評価の議論

こうしたアプローチは、伝統的なLispコミュニティからしばしば「古めかしい設計の誤りを繰り返している」との批判に晒されてきた 。

批判の主な論点には以下のものが挙げられる。

* 動的スコープの強制や、コンパイル機構の欠如（インタプリタ実行のみ）による実行速度の低下 。
* 値の受け渡しのたびに発生するディープコピーによるメモリ効率の懸念 。
* 第2引数がリストでない場合の `cons` 関数が `list` と同様に振る舞うなど、Lispの伝統的なデータ構造（ドットペアなど）との非互換性 。
* fexpr（引数を評価せずに受け取る関数）や自己書き換えコード（Self-modifying code）の多用が、近代的な計算機科学のパラダイムから逸脱しており、予期せぬバグの温床になりやすい点 。


これに対し、newLISPの擁護派やスクリプト開発者は、言語としての「実用性」と「敷居の低さ」を最大の強みとして評価している 。

* Perlやbash、AppleScriptのような日常的なシェルタスク、CGIスクリプトとしての立ち上がりの早さは比類がない 。
* データベースインターフェース（SQLite、MySQL、PostgreSQL、ODBC）、TCP/IPおよびUDPネットワーク、正規表現、XML、セマフォによるマルチプロセス、UTF-8マルチバイト文字への対応、Bayesian統計など、現代のスクリプト開発に必要なツールが追加パッケージの導入なしにデフォルトで「ビルトイン（電池付属）」されている 。
* 言語コアが非常にシンプルであるため、C言語の関数を直接バインドする `import` が非常に低オーバーヘッドで機能する 。

このような議論の場として機能していた「newLISP Fan Club」（`newlispfanclub.alh.net`）は、ドメイン閉鎖後もファンや開発者が集う非公式のフォーラムであり、ライブラリ接続の技術支援や代替ソースの共有、Emscriptenを利用したブラウザ上への移植プロジェクトの議論など、知的アーカイブの蓄積に貢献している 。
