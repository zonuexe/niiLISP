# 実装レトロスペクティブ: Rust 選定の見積もりと実装後の実相 (2026-07-04)

v0.1.0 までを実装した時点での振り返り。[ADR-0004](../adr/0004-implementation-language-rust.md) で実装言語に Rust を選び、[ADR-0005](../adr/0005-vec-backed-lists.md)/[0006](../adr/0006-dynamic-scope-mechanism.md)/[0011](../adr/0011-error-model.md)/[0013](../adr/0013-string-representation-and-unicode.md) でメモリ・スコープ・エラー・文字列の方針を立てた。実装を経て、その見積もりがどこで当たり、どこで外れ、何を新たに学んだかを記録する。

## 1. Rust 選定時の見積もり (ADR-0004)

- **相性◎の根拠**: ORO（One Reference Only）が Rust の所有権に 1:1 で対応する。単一所有＝所有権、渡すときのディープコピー＝`clone`、スタック的解放＝`Drop`、そして「循環参照が生じない」を借用規則が**型で保証**する。
- **懸念**: グローバルに参照される構造（context / symbol table、インターンされたシンボル、Lisp オブジェクトグラフ）が借用検査と衝突し、arena / `RefCell` / 局所的 `unsafe` が要るだろう。
- 学習コストは高いが優先度最下位なので許容。

## 2. 実装後の実相

### 2.1 見積もり通りだったもの

- **Vec バックのリスト (ADR-0005)**: `Value::List(Vec<Value>)` を「格納・受け渡し時に `Clone`」する設計が、そのまま ORO のディープコピーになった。`Str(Vec<u8>)` も同様。素直で、特別な工夫は要らなかった。
- **動的スコープ (ADR-0006)**: シンボルごとの値スロット（`RefCell<HashMap<SymId, Value>>`）＋ `Scope` という `Drop` ガードによる save/restore。予測どおりに実装でき、しかも**巻き戻し安全（早期 return・エラーでも復元）が `Drop` で無償**についてきた。
- **エラーモデル (ADR-0011)**: `eval` が `Result<Value, Signal>` を返し、`?` で伝播、`catch` で捕捉。ここが最も的中した予測で、「エラー巻き戻し・動的スコープ復元・ORO 解放」が **`?` と `Drop` の一本道**に統一された。別々の後始末を書く必要がなく、正しさが `Drop` の順序に乗った。
- **`Vec<u8>` 文字列 (ADR-0013)**: バイナリ安全なバイト列として素直に実装。`length` はバイト数、という newLISP 挙動もそのまま出た。

### 2.2 見積もりより良かった点（嬉しい誤算）

- **借用検査との衝突が、ほとんど起きなかった。** 理由は ORO そのものにある。ORO は共有を禁じるので、Lisp のオブジェクトグラフは「共有された可変グラフ」ではなく**所有された木**になる。Rust が最も苦手とする「共有可変グラフ」という難所が、ORO の無共有性によって**構造的に存在しなくなっていた**。ADR-0004 が恐れた摩擦の大半は、ORO と Rust の相性がそれを先に消していた。相性は見積もり以上だった。
- **`unsafe` はゼロ。** ADR-0004 は「局所的 `unsafe` が要るかも」としたが、最後まで一箇所も要らなかった。インタプリタ全体が安全 Rust で書けている。
- **arena は不要だった。** context を独立したメモリ領域（arena）で持つ想定だったが、実際は **`Ctx:member` という修飾名をインターンした平坦な `HashMap`** で足りた。context は別構造ではなく「名前空間つきシンボルの集合」に還元でき、`Value::Context(SymId)` は名前を指すだけの薄い値になった。

### 2.3 Rust が強いた、唯一の本質的な設計転換

見積もり時に具体化できていなかった最大の適応点がこれ。

- **「関数から返り、その後の評価を跨いで保持される参照」は Rust の `&mut` では表現できない。** 借用検査は、可変借用を任意のコード実行を跨いで保持することを許さない。ところが newLISP の ORO 参照モデル（[qa-ref](../../references/newlisp/qa-specific-tests/qa-ref)）は、`(pop (sort s))` や `(replace 'f (nth 2 L) 'z)` のように、**式が「格納構造の中への参照」を返し、後続の破壊的操作がそれを書き換える**ことを要求する。newLISP(C) はセルへの生ポインタでこれを実現している。
- niiLISP はこれを **「参照を*データ*として符号化する」** ことで解いた。参照＝`Place { root: SymId, path: Vec<i64> }`（根のシンボルと、そこへのリスト添字の経路）という**所有された値**であり、書き込みの直前に `place_navigate` で `&mut Value` へ再ナビゲートする。`resolve_place` は `if`/`when`/`begin`/`and`/`or`/`cond`/`case`/`set`/`assoc`/`sort`/`rotate` などの制御構文・破壊的関数まで再帰して place を返す。`self` も `Vec<Place>` の symbolic-place スタックとして持つ。
- 要するに、**ポインタ意味論を「経路データ」で再現した。** これにより、安全 Rust のまま newLISP の参照/place モデル（self write-back、破壊的操作の連鎖）を満たせた。これは選定時に見えていなかった、実装で最も頭を使った転換点であり、同時に「Rust で newLISP をやる」ことの核心的な回答になった。

### 2.4 ORO からの意図的な逸脱

- **コード値は `Rc<Lambda>` で共有している。** 純 ORO なら関数値もディープコピーすべきだが、シンボルを引くたびに body を丸ごとコピーするのは無駄で、自己書き換えコードは v1 の対象外なので、lambda/fexpr は `Rc` 共有にした。
- 帰結として**小さな穴**が残る: 自己書き換え（qa-inplace は関数本体のリテラル `0` を破壊的に書き換える）を実装するなら、`Rc` の共有 body に対して `Rc::make_mut`（copy-on-write）で ORO の単一所有を回復する必要がある。データ（リスト・文字列）は完全に ORO だが、コードだけは参照カウント、という非対称を承知の上で置いている。

## 3. メモリモデルの考察

- **ディープコピーのコスト**: ORO は受け渡し毎に O(n) のコピーを払う。newLISP と同じ挙動で、優先度①「互換」・correctness-first（[ADR-0007](../adr/0007-evaluator-tree-walk-with-cache.md)）の下では許容。むしろ Vec バックなので連続メモリの `clone` になり、cons セル鎖を辿るより**キャッシュ効率は良い可能性がある**。ADR-0007 が予告した派生キャッシュ（`a'`）はまだ未実装で、速度が問題化してからで良い。
- **GC 不在**: `Drop` が決定的な解放を担い、ORO の「評価階層の巻き戻しと同時にスタック的に解放」と一致する。ADR-0011 の「巻き戻し規律の一本化」は机上の主張ではなく、実装で実際にそうなった。
- **循環参照**: ORO が構造的に排除する。唯一の `Rc`（Lambda）も自己参照を作れないのでリークは生じない。ただし ADR-0004 の当初の言い回し「循環不能を**借用検査が**型で保証する」は、実装後の目で見ると**不正確**だった。実際に循環を防いでいるのは「Lambda 以外を全部ディープコピーする設計規律」であって、`Rc` を導入した時点で借用検査は循環防止の主役ではない。相性の説明としては正しかったが、保証の主体を取り違えていた。
- **単一スレッド不変条件 (ADR-0014)**: fork ベース並行と callback トランポリンの前提。`RefCell` 中心（`Send`/`Sync` 不要）の設計はこれと整合的。将来スレッドを入れるなら、fork-and-continue を壊さないことがハードな制約になる。

## 4. `RefCell` 運用の教訓

- 可変状態（globals / interner / self_stack / protected）はすべて `RefCell` に置いた。`RefCell` の実行時借用パニックを避けられた鍵は **「評価を跨いで borrow を保持しない」規律**にある: place の添字や値は先に `eval` して確定させ、`with_place_mut` は書き込み直前に**一瞬だけ** `borrow_mut` する。`read_place` も同様に短命。
- これは 2.3 の symbolic-place 方式と同じ発想（参照を保持せず、データ化して必要な瞬間に解決する）であり、借用検査と `RefCell` の両方を同時に穏当にした。設計の一貫性がそのまま実装の平穏につながった。

## 5. 総括

- 選定時の見積もり（Rust ↔ ORO は好相性、ただし借用検査との摩擦あり）は、**方向は正しく、程度を外した**: 相性は予想以上、摩擦は予想以下。ORO の無共有性が「共有可変グラフ」という Rust 最大の難所を先に消していたためで、`unsafe` ゼロ・arena 不要・オブジェクトグラフの借用衝突なし、で済んだ。
- 代わりに、見積もりで具体化していなかった **「参照をポインタではなくデータ（symbolic place）として表す」** が最大の適応点となり、これが安全 Rust を保ったまま newLISP の参照モデルを再現する橋になった。ここが本プロジェクトで最も Rust らしい判断だった。
- ORO からの逸脱は `Rc<Lambda>` の一点のみ。自己書き換えコード対応時に copy-on-write で塞ぐ、という宿題として明示的に残す。
- 結論として Rust 選定は妥当だった。ADR-0004 の次点（Zig/Nim）や見送り（MoonBit、GC 系）と比べ、**安全性・`Drop` による巻き戻しの一本化・`unsafe` ゼロ**という実装後の実利は、選定時の想定を上回った。GC 系言語を選んでいたら「GC ランタイムの上に no-GC 意味論（ORO）を載せる」倒錯になっていたはずで、そこを避けられたのは大きい。
