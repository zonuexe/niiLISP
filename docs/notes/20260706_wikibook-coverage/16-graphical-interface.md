# Ch. 16 — Graphical interface

The chapter teaches newLISP-GS: load `guiserver.lsp`, `gs:init` a Java Swing subprocess over two localhost sockets, build a widget tree, and `gs:listen` an event loop; niiLISP does not run that stack at all, but its default binary already contains every non-GUI prerequisite the stack depends on, and a `gs:`-vocabulary module (`lib/gui.lsp`) is vendored and talks the same base64/line protocol to an opt-in native `niilisp-gui` (fltk) helper instead of a JVM.

## niiLISP's divergent GUI approach

The WikiBook chapter's entire workflow assumes **newLISP-GS**: `guiserver.lsp` (a `.lsp` module bundled with stock newLISP) shells out to `java -jar guiserver.jar <port>`, a Java Swing process, and drives it over two localhost TCP sockets (base64-encoded, newline-terminated command lines out; `eval-string`'d newLISP source lines in as events). None of that JVM machinery exists in niiLISP, and it never will — there is no plan to vendor a `guiserver.jar` or require a JVM on PATH.

Instead, per **ADR-0034** ("A native GUI: an fltk helper process driven over a socket (gs:-inspired)"), niiLISP took the position that it already has the *language* substrate newLISP-GS depends on — `process`, `net-*`, `base64-*`, `eval-string` — so rather than ship Java, it ships:

- **`niilisp-gui`** — a second Rust binary (`src/bin/niilisp-gui.rs`) using `fltk-rs`, a native retained-mode toolkit with no JVM/GPU dependency, bundled and cross-platform. It owns its own main thread (required by e.g. macOS Cocoa) and is launched as a subprocess, exactly mirroring the "two processes talking over a socket" shape of the original guiserver — just with fltk in place of Swing.
- **`lib/gui.lsp`** — a vendored `.lsp` module (verified present and loadable) that defines `gs:frame`, `gs:panel`, `gs:button`, `gs:label`, `gs:text-field`, `gs:add-to`, `gs:set-text`, `gs:set-background`, `gs:init`, and command-building helpers (`gs:line`, `gs:enc`). Loading it and calling `gs:frame-cmd "F" 10 10 100 100 "Hello"` (without a live socket) correctly produces the protocol line `frame F 10 10 100 100 SGVsbG8=` — confirming the sender/encoding half works standalone. The vocabulary and general shape (id-based retained widgets, base64-encoded text tokens, line-oriented commands) intentionally track `gs:` closely "so existing `gs:` scripts run with as little rewriting as possible" (ADR-0034), but with **no behavior guarantee** — a different toolkit will not reproduce Swing's exact pixels, fonts, or layout math.
- **Packaging**: both the `fltk` dependency and `niilisp-gui` binary sit behind a `gui` Cargo feature that defaults **off** (`default = ["ffi", "bigint", "regex", "mt", "net", "readline"]` — `gui` is conspicuously absent). The default `niilisp` interpreter binary never links fltk. In this checkout, `target/release/niilisp` has no accompanying `niilisp-gui`; only a `target/debug` build (evidently built once with `--features gui` for development) has one, confirming the feature is genuinely opt-in and not part of what ships by default.

Net effect: the book's exact recipe (`(load (append (env "NEWLISPDIR") "/guiserver.lsp"))` → `(gs:init)` → JVM window) cannot and will not work, ever, in niiLISP. But an architecturally analogous, differently-branded recipe (`(load "lib/gui.lsp")` → `(gs:init)` → native fltk window) is under active construction and partially working today, gated behind an explicit build flag.

## Divergences & gaps

- **No JVM path exists or is planned.** `gs:init` as the book describes it is not a gap to be closed — it is out of scope permanently. Any coverage of `gs:*` names in niiLISP is coverage of a *lookalike* API, not the original.
- **The `gui` feature is not in the default build**, so a stock `niilisp` binary — the one this whole audit otherwise probes — cannot open a window at all today. `(gs:frame)` on an unmodified REPL errors `not a function: nil` because the `gs` context is never created; it only exists after explicitly `(load "lib/gui.lsp")`, and even then `gs:init` needs a `niilisp-gui` executable on PATH (or `NIILISP_GUI` set) that a default `cargo build`/release binary does not produce.
- **`lib/gui.lsp`'s widget/layout coverage is a first slice, not the full `gs:` surface.** Confirmed present: `gs:init`, `gs:frame`/`gs:window`, `gs:panel`, `gs:button`, `gs:label`, `gs:text-field`, `gs:add-to`, `gs:set-text`, `gs:set-background`. Confirmed absent from the module as vendored: layout managers (`gs:set-border-layout`, `gs:set-grid-layout`), `gs:slider`, `gs:canvas` + drawing, `gs:set-resizable`, `gs:set-visible`, `gs:set-color`, `gs:set-font`. ADR-0034 explicitly scopes canvas/drawing to a later slice, and other book-used widgets/layouts beyond the "first slice" list are simply not built yet.
- **The prerequisite substrate is the pleasant surprise.** Contrary to a naive assumption that niiLISP "doesn't do GUI," every non-visual building block newLISP-GS itself depends on — `load`, `env`, `process` (non-blocking launch), the six core `net-*` calls, `base64-enc`/`base64-dec`, and `eval-string` — is implemented and functionally verified in the *default* binary (no `gui` feature needed). This is precisely the dependency chain the existing gap-analysis note (`docs/notes/20260706_newlisp-gap-analysis.md`) had flagged as the GUI chapter's real blocker; it is now closed. The only remaining blocker to any windowed output is the `gui` feature flag and the fltk toolkit itself, not the language substrate.
- **No end-to-end verification is possible or attempted here.** Per this audit's constraints (and per ADR-0034's own stated acceptance criteria), the GUI is display-dependent and not something to launch or block on; only the protocol/sender layer (`gs:frame-cmd` etc., pure functions with no socket) was exercised, matching ADR-0034's own CI strategy of unit-testing the protocol layer while leaving the real window to manual human verification.
- **Behavioral parity is explicitly disclaimed, not just unimplemented.** Even where `gs:*` names exist and eventually render via fltk, ADR-0034 states there is "no behaviour guarantee" versus Swing — so this chapter's coverage ceiling, even at 100% of the vendored slice, is a differently-shaped GUI, not a compatibility target.
