# newLISP という言語の特異性 — 現代的インタプリタ実装・主要 Lisp 仕様とのギャップから読む (2026-07-04)

niiLISP を実装する過程で見えた newLISP の設計思想を、(1) 現代的な「王道」インタプリタ実装、(2) 主要な Lisp 族言語の仕様、との差分として紹介するノート。多くの特異性が **ORO（One Reference Only）というただ一つの公理**から芋づる式に導かれる、という視点で整理する。実装で実際に触れた事実は該当箇所に注記した。

なお本ノートは newLISP を貶めるためのものではない。むしろ「一貫した公理から言語全体を導く」という設計の潔さを、標準的な選択との差として浮かび上がらせるのが狙いである。

---

## 1. 現代的な「王道」インタプリタ実装とのギャップ

近年の動的言語処理系（『Crafting Interpreters』的な教科書、および Lua / CPython / 各種 JS エンジン）が収斂してきた「王道」と、newLISP の選択を並べる。

| 論点 | 現代の王道 | newLISP | 帰結 |
| --- | --- | --- | --- |
| 実行方式 | ソースを**バイトコードへコンパイル**し VM で実行 | 生の S 式（ライブなリスト構造）を**直接 tree-walk** | 自己書き換えコードが可能な代わりに低速 |
| メモリ管理 | トレーシング GC（世代別）／サイクル検出付き参照カウント | **ORO**: 共有せず、格納・受け渡しでディープコピー、巻き戻しで即時解放、GC なし | ~250-300KB バイナリ・瞬時起動・GC ポーズ皆無 |
| データ共有 | 積極的に共有（interning、構造共有、永続データ構造、CoW） | **一切共有しない** | 循環参照が原理的に発生しない |
| 値表現 | NaN-boxing／ポインタタグ付けで小さく | 均一な固定長セル | 単純だが省メモリ最適化は限定的 |
| スコープ | **レキシカル＋クロージャ**（upvalue／環境） | **動的スコープ、クロージャなし** | 実行時の柔軟性 vs 静的解析の困難 |
| オブジェクト | 参照渡し | **値渡し（コピー）**＋参照系の後付け | 下記「補償レイヤ」を要する |
| マクロ | 衛生的マクロ（`syntax-rules` 等） | **fexpr**（実行時・未評価引数・非衛生） | コンパイル不能と表裏一体 |
| 末尾呼び出し | TCO を実装することが多い | **TCO なし**（ホストスタックを使い最大深度で打ち切り） | 深い再帰はスタック上限で失敗 |
| 高速化 | インラインキャッシュ／JIT | なし | 「実用十分」に振る |

### 特に効いている 3 点

- **バイトコード不採用は「意図」である。** newLISP はコードをデータとして生のまま保持し、実行時に破壊的に書き換えられる（[qa-inplace](../../references/newlisp/qa-specific-tests/qa-inplace) は関数本体のリテラルを書き換える）。バイトコードへコンパイルすると「コード＝ライブデータ」でなくなり、この能力と fexpr が壊れる。**fexpr・自己書き換え・tree-walk は一つのクラスタ**を成し、どれか一つを採ると残りが付いてくる。
- **ORO は GC の代替ではなく、GC を「不要にする」設計。** 共有を禁じれば循環が生じず、循環が生じなければ到達可能性の走査（＝GC の本質）が要らない。解放は評価階層の巻き戻しと同時にスタック的に行える。GC を高度化する現代の潮流に対し、newLISP は**問題そのものを消す**方向に賭けた。
- **値表現の均一セル**は、この巻き戻し解放とフリーリスト運用を単純にするための選択。（niiLISP は ORO の意味論だけ再現し、内部は `Vec` バックのリストにした。[ADR-0005](../adr/0005-vec-backed-lists.md)）

## 2. 主要 Lisp 族言語の仕様とのギャップ

Common Lisp (CL) / Scheme / Clojure / Emacs Lisp と並べると、newLISP は「Lisp の名を持つが Lisp の根幹の幾つかを持たない」ことが分かる。

### 2.1 cons とドットペアがない — 最も根源的な断絶

伝統的 Lisp の土台は **cons セル（car / cdr）とドットペア** `(a . b)` である。連想リスト・非真リスト・木構造すべてがこの上に載る。

newLISP の `cons` は、第2引数がリストでないと `list` と同じに振る舞う: `(cons 'a 'b)` → `(a b)`（`(a . b)` ではない）。**ドットペアが存在しない。** これは「実装の癖」ではなく、Lisp のもっとも基礎的なデータ構造を捨てたということ。(niiLISP はこれを逆手に取り、cdr が任意値を指せない前提で `Vec` バックのリストを正当化した。[ADR-0005](../adr/0005-vec-backed-lists.md))

### 2.2 動的スコープ、クロージャなし

CL は既定レキシカル＋特殊変数で動的、Scheme / Clojure はレキシカルのみ、Emacs Lisp は歴史的に動的（現在はレキシカル選択可）。**newLISP は動的スコープのみ**で、レキシカルクロージャを持たない。状態を閉じ込めたければ context を使う。「クロージャ中心」の現代 Lisp／関数型の常識から最も外れる点。

### 2.3 同一性 `eq` の消失 — ORO の深い帰結

Lisp は **cons セルの同一性（`eq`）** の上に築かれている（CL の `eq`/`eql`/`equal`/`equalp`、Scheme の `eq?`/`eqv?`/`equal?`）。ところが ORO では**あらゆるデータ値がコピーされる**ので、「同じオブジェクトか」という同一性が**そもそも意味を失う**。共有が無い世界に持続的アイデンティティは無い。newLISP に実質あるのは構造的等価 `=` だけである。(context やシンボルはグローバルに一意なので同一性を持つが、データ値は持たない。) これは Lisp の根本前提の一つが ORO によって静かに消えている、という驚くべき事実。

### 2.4 fexpr の復活 — Lisp が捨てたものを拾い直す

fexpr（引数を評価せず受け取る関数）は 1980 年前後に Lisp から意図的に排除された（コンパイルと静的解析を破壊するため；Pitman の "special forms" の議論）。以後の主流は**衛生的マクロ**（コンパイル時展開）である。newLISP はこれを `lambda-macro` / `define-macro` として**実行時 fexpr のまま復活**させた。歴史的には最も論争的な選択だが、newLISP がそもそもコンパイルしない（tree-walk）ので**矛盾はない** — fexpr は純インタプリタでのみ筋が通る。(CONTEXT.md の "fexpr" 参照)

### 2.5 context の多重役割 — 名前空間の overload

CL のパッケージ、Scheme/Racket のモジュール、Clojure の namespace は「名前空間」に専念する。newLISP の **context は、名前空間・プロトタイプベースのオブジェクト（FOOP）・辞書（ハッシュ）・モジュールを一つの機構で兼ねる**。強力だが概念的に過積載で、`Ctx:member` という修飾シンボルと default functor、コロンディスパッチが一体で動く。(niiLISP では context を「`Ctx:name` をインターンした平坦なシンボル集合」に還元できた。[ADR-0010](../adr/0010-foop-object-model.md))

### 2.6 その他の仕様差（一覧）

| 論点 | CL | Scheme | Clojure | newLISP |
| --- | --- | --- | --- | --- |
| ドットペア/真の cons | あり | あり | (seq 抽象) | **なし** |
| スコープ | 既定レキシカル | レキシカルのみ | レキシカル | **動的のみ** |
| クロージャ | あり | あり | あり | **なし**（context で代替） |
| マクロ | 非衛生マクロ | 衛生マクロ | マクロ | **fexpr** |
| 末尾呼び出し | 実装依存 | **必須(TCO)** | recur | **なし** |
| 継続 | — | **call/cc** | — | なし |
| 数値タワー | 完全（有理・複素・bignum） | 完全 | 有理・bignum | **int64 + double + bigint(L) のみ** |
| 等価 | eq/eql/equal/equalp | eq?/eqv?/equal? | =/identical? | **`=` のみ**（同一性は消失, 2.3） |
| 準クォート | backquote | backquote | syntax-quote | なし（`expand`/`letex` で代替） |
| 条件系 | **再起動可能な condition** | ガード/例外 | ex-info | **catch/throw + error-event** のみ |
| 偽値 | `nil`(=空リスト) | `#f`（空リストは真) | `nil`,`false` | `nil` と `()` がともに偽 |

## 3. 総合: ORO を公理とした特異性の体系

newLISP の「変わっている」ところは、バラバラの奇癖ではなく、**「値を共有しない（ORO）」という一つの公理**からほぼ演繹できる。

```
ORO（共有しない・ディープコピー）
├─ 循環が生じない ──→ 到達可能性走査が不要 ──→ GC 不要 ──→ 極小バイナリ・瞬時起動・ポーズ皆無
├─ 同一性が消える ──→ eq が無意味 ──→ 構造的 = のみ（2.3）
├─ 参照渡しができない ──→ 補償レイヤが必要:
│     ・context を「名前」で渡す（値でなく）
│     ・default functor / コロンディスパッチ / self
│     ・破壊的操作のための「参照/place 系」(setf, nth-as-place, replace…)
└─ 値意味論が全面化 ──→ FOOP オブジェクトは「タグ付きリストのコピー」＋
                          後付けの参照機構で変更を書き戻す
```

これに、コンパイルを捨てた系（**fexpr ＋ 自己書き換え ＋ tree-walk** のクラスタ、§1）と、**動的スコープ＋クロージャ不在 → context による一元化**（§2.2, 2.5）が重なって、言語全体の姿が決まる。

### 実装で最も痛感した「補償レイヤ」

ORO が参照を消したので、それを取り戻すための機構が newLISP には後付けで大量にある。niiLISP 実装で最も複雑だったのは、まさにこの層だった:

- **参照/place 系**（[qa-ref](../../references/newlisp/qa-specific-tests/qa-ref)）: `(pop (sort s))` や `(replace 'f (nth 2 L) 'z)` のように、**式が構造の中への参照を返し**、破壊的操作がそれを書き換える。値意味論の言語に「参照」を接ぎ木する、最も newLISP らしい発明。
- **self の write-back**（[qa-foop](../../references/newlisp/qa-specific-tests/qa-foop)）: メソッド内の `(inc (self 1))` が対象オブジェクトを実際に書き換える。self は値のコピーではなく参照でなければならない。
- niiLISP ではこれらを「参照＝経路データ（symbolic place）」として表現した（[レトロスペクティブ](20260704_rust-oro-retrospective.md) §2.3）。つまり **ORO が値意味論を敷いた上に、newLISP は参照意味論を島のように再導入している**。この二層構造こそ newLISP の複雑さの源泉であり、同時にその独自性の中心でもある。

### 思想: 理論的純度より実用スクリプティング

これらの選択は「Lisp 理論家を満足させる」ためではなく、**Perl / bash / AppleScript のように瞬時に立ち上がる自己完結スクリプト言語**を作るための最適化である（ルッツ・ミュラーの設計目標）。SQLite・ネットワーク・正規表現・統計まで「電池付属」で、`import` の低オーバーヘッド、極小バイナリ、瞬時起動 — これらは ORO の設計とセットで、実用ツールとしての強みになっている。

### 賛否

- **批判**（伝統的 Lisp コミュニティ）: 動的スコープの強制、fexpr の多用、TCO 欠如、コンパイル機構の不在による低速、ディープコピーのメモリ効率、`cons` がドットペアを壊す — 「古い設計の誤りを繰り返している」。
- **擁護**: 敷居の低さと実用性、`import` の軽さ、そして何より **ORO という一貫した独自メモリモデル** の潔さ。GC を高度化するのでなく不要にする、という発想は 2020 年代の視点でもむしろ新鮮に映る。

## 4. 実装者としての一言

「Lisp とは何か」を規定する要素（cons と `eq`、レキシカルスコープとクロージャ、衛生的マクロ、TCO）を newLISP はことごとく持たない。にもかかわらず、S 式・quote・第一級リスト・シンボルという**表層の Lisp らしさ**は保っている。newLISP は「Lisp の構文と、まったく別の意味論的公理（ORO）」を接合した処理系であり、その意味で **Lisp 族の中の変種というより、S 式を借用した別系統の言語**と見るのが実装者としての実感に近い。niiLISP の設計判断（[docs/adr/](../adr/)）の大半が「標準的な Lisp 実装ならこうする」を一度括弧に入れ、「ORO ならどうなるか」を問い直す作業だったのは、この特異性の直接の反映である。
